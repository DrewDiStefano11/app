# 07. Insight Safety, Privacy, and User Control

## Safety and Privacy Expectations

Insights generated by FitCore must respect user privacy and adhere to strict safety boundaries. Sensitive health, medical, or genetic data requires extra controls and explicit consent before being utilized for analytics or AI recommendations.

### Privacy-Safe Aggregation

*   Medical, genetic, injury, pain, symptom, photo, and conversation-derived insights require stronger consent than basic workout logs.
*   Dashboards must not expose sensitive information too casually (e.g., prominently displaying a sensitive health condition on the main summary screen).
*   Future implementation of a privacy mode should reduce or hide sensitive insight cards.
*   Exported insight reports must allow users to choose which categories of data are included.

## Category-Based AI Use Controls

Users must have granular control over what data categories the AI is permitted to use for generating insights.

Users can toggle AI access for:
*   Training data
*   Nutrition data
*   Recovery data
*   Body metrics
*   Medical history (From Book 8)
*   Injury history (From Book 8)
*   Genetic/lab data (From Book 8)
*   Photos/conversations (where applicable)

### Human Review and User Override

The user remains in absolute control of their data and insights. Users must be able to:
*   Hide specific data from insights.
*   Delete specific data points (which must subsequently be removed from Health Twin influence).
*   Correct inaccurate data.
*   Disagree with insights or dismiss irrelevant recommendations.
*   Export insight-related source data.
*   Turn off certain AI memory categories entirely.
*   Choose not to use AI-generated insights at all.

FitCore should not punish the user for skipping logs, ignoring a recommendation, or opting out of AI features.

## Red-Flag Boundaries

Insights must **never diagnose or replace medical care**. The AI must adhere to these red-flag boundaries:
*   FitCore may recommend professional care when symptoms look concerning (e.g., "Your pain logs indicate worsening sharp pain in your knee; consider consulting a physical therapist").
*   FitCore **should not** diagnose the cause (e.g., *Never* say "You have a torn meniscus").
*   FitCore must use cautious, non-definitive language for any health-related insights.

## Insight Priority and Ranking

Insights must be prioritized intelligently to avoid overwhelming the user and to ensure critical information is seen first.

*   Urgent safety-related insights (e.g., worsening pain trends) rank higher than normal progress insights.
*   Major negative trend changes rank higher than minor positive improvements.
*   Stale or low-confidence insights should not dominate the dashboard.
*   Repetitive insights should be suppressed or combined.
*   User goals influence insight priority (e.g., protein adherence ranks higher during a muscle-gain phase).
*   The system should not overwhelm the user with too many cards at once.

**Priority Examples:**
*   Pain increasing across multiple workouts appears *before* a minor calorie consistency note.
*   Poor sleep combined with high soreness appears *before* a generic workout streak message.
*   A one-day weight spike should not be treated as a major issue and should rank low or be suppressed entirely.

## Insight Lifecycle and Refresh Rules

Insights follow a specific lifecycle:
1.  **Detected:** The underlying system identifies a pattern.
2.  **Generated:** The AI formulates the insight and recommendation.
3.  **Shown to user:** The insight appears on a dashboard.
4.  **Explained:** The user requests the "why" behind the insight.
5.  **Actioned:** Accepted, dismissed, corrected, or ignored by the user.
6.  **Archived/Replaced:** Removed when no longer relevant.
7.  **Feedback Loop:** Used to improve future recommendations (if permitted by the user).

### Insight Refresh Rules

Insights should be refreshed dynamically, but not so frequently that the dashboard feels unstable. Refresh triggers include:
*   After new logs are added that significantly alter the current data picture.
*   After user corrections to source data.
*   After data deletion.
*   After goal changes.
*   After a new week or month begins (for time-bound insights).
*   After enough new data changes a recognized trend.
*   After imported data syncs (in future phases).

FitCore must avoid showing outdated insights after the underlying data changes. Old insights should not stay visible forever if the pattern has resolved.

## Insight Versioning and Auditability

To maintain trust and support debugging, insights must be auditable. The system should track:
*   When the insight was generated.
*   What source data was used to generate it.
*   Whether that source data has changed since generation.
*   Whether the user corrected any source data.
*   Whether the insight should be refreshed based on new data.
*   Whether the insight was previously dismissed or marked wrong by the user.

## Insight Acceptance Criteria (Planning Level)

For future implementation of the analytics engine, the following acceptance criteria apply:
*   Every major score has a verifiable explanation.
*   Every major insight can display its source data category.
*   Stale data is visually labeled.
*   Missing data is explicitly noted, not hidden behind false calculations.
*   User-corrected values successfully update future insights.
*   Deleted data no longer influences newly generated insights.
*   Dismissed insights do not immediately reappear unchanged.
*   The AI respects category-based use controls (does not use disabled data).
*   Graphs and dashboards handle empty accounts gracefully without errors or fake data.
*   Insights do not make medical diagnosis claims.
*   Sensitive insight categories correctly require stronger privacy controls.

## Testing Expectations for Future Implementation

**Note: This is planning only. Do not implement tests in this phase.**

Future implementation phases must include automated and manual tests to verify the following:
*   Score explanations accurately match the underlying source data.
*   Graph mode persistence works correctly across navigation.
*   Stale data warnings appear correctly based on defined thresholds.
*   Dismissed insights remain dismissed according to lifecycle rules.
*   Corrected data triggers a recalculation and updates relevant trends.
*   The AI definitively ignores data categories disabled by the user.
*   Deleted data no longer influences any active insights.
*   Low-confidence insights are correctly labeled in the UI.
*   Graphs and dashboards render correctly without errors for empty accounts.
*   Duplicate data inputs do not create outsized or misleading insights.
