# FitCore Data Trust Implementation Plan

## 1. Visual Language for Data Provenance

To build trust, every piece of data in FitCore must disclose its origin and certainty.

### Badge Naming & Definitions

| Badge Label  | Data Source       | Definition                                                    |
| :----------- | :---------------- | :------------------------------------------------------------ |
| **Manual**   | User-entered      | High-trust; entered via form or numeric input.                |
| **Verified** | User-corrected AI | Highest-trust; AI data that a user has reviewed and saved.    |
| **Jarvis**   | AI Assistant      | Moderate-trust; data generated by text-to-log or voice.       |
| **Camera**   | AI Vision         | Variable-trust; data estimated from a photo.                  |
| **Imported** | JSON/Restored     | High-trust (usually); data brought in from a previous backup. |
| **Sensor**   | Wearable/Health   | Moderate-trust; raw data from external devices (future).      |

### Confidence States

- **High Confidence (Green):** Manual entries, Verified AI, or high-probability sensor data.
- **Medium Confidence (Amber):** AI text estimates with clear entities.
- **Low Confidence (Red/Dashed):** Unclear photos, vague text, or partial sensor data.

---

## 2. UI Placement Rules

### Where Badges Appear

1. **Nutrition History:** Next to meal item names in the daily list.
2. **Workout History:** On the exercise header within the summary view.
3. **Detail Sheets:** In the "Metadata" section at the bottom of the sheet.
4. **Dashboard Tiles:** Only show "Low Confidence" warnings (e.g., a dashed ring around a macro tile).

### Hierarchy & Overrides

- **Priority:** `Verified` > `Manual` > `Sensor` > `Jarvis` > `Camera`.
- **User Action:** If a user edits _any_ value in an AI-generated log, the badge must immediately upgrade to `Verified`.
- **Fallback:** For legacy data missing provenance, default to `Manual` but without a visible badge to avoid cluttering old history.

---

## 3. Trusted Engine Integration

The Daily Decision Engine uses these trust levels to weight recommendations:

- **Low Confidence Impact:** If 50% of today's nutrition data is `Low Confidence` (Camera estimates), the Engine should soften its "Deficit/Surplus" claims and add a "Data Quality" warning.
- **Missing Data:** If mandatory fields are missing (e.g., no weight logged in 7 days), the recommendation should prioritize "Log Weight" over "Change Volume".

---

## 4. Implementation Stages

### Stage 1: The "Audit Row" (Jules)

- Add a subtle `Eyebrow` or small `Tile` at the bottom of every detail sheet (Meal/Workout) showing:
  - Source Badge
  - Timestamp of creation
  - "Last Verified" date

### Stage 2: List Visibility (Jules)

- Add 12px icon badges to history list items.
- Use distinct icons for Camera (📷), Jarvis (🤖), and Manual (✍️).

### Stage 3: Low-Confidence Warnings (Jules)

- Implement a `low-confidence` Tailwind class that adds a subtle amber glow or dashed border to cards/tiles containing unverified estimates.
