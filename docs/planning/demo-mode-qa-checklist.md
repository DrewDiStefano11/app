# Demo Mode QA Checklist

## 1. Purpose
The purpose of this checklist is to verify that Demo Mode functions correctly as an isolated, safe environment for users to explore FitCore's features. It ensures that demo data never bleeds into real user accounts and that the demo experience accurately reflects the core product value.

## 2. Scope
This checklist covers the lifecycle and boundaries of Demo Mode:
* Demo mode entry
* Demo account identification
* Demo seed data labeling
* Demo data separation from real data
* Demo dashboards
* Demo graphs
* Demo AI/Jarvis behavior
* Demo reset behavior
* Preventing demo data from polluting real accounts
* First tester safety checks

## 3. When to use this checklist
* Whenever modifying the app's initialization or state hydration logic.
* After changing the database schema or local storage keys.
* Prior to marketing pushes where Demo Mode will be heavily utilized by new users.

## 4. Required preconditions
* Use a fresh installation or clear local storage before starting.
* Ensure you have a real account to test the boundaries between demo and real data.
* Familiarity with how demo state is managed (e.g., using `useStore().view` vs `useStore().state`).

## 5. Step-by-step checklist
1. **Demo mode entry**: Launch the app without logging in and select "Try Demo". Verify the app enters demo mode quickly and smoothly.
2. **Demo account identification**: Check that the UI clearly indicates the app is currently in demo mode (e.g., a banner or badge).
3. **Demo seed data labeling**: Verify that pre-populated demo data (meals, workouts) is clearly labeled, and not mistaken for user-generated data.
4. **Demo dashboards**: Navigate through the home dashboard and verify all widgets populate with the seeded demo data.
5. **Demo graphs**: Open history graphs and ensure they render correctly using the demo data points.
6. **Demo AI/Jarvis behavior**: Interact with Jarvis. Verify it uses the demo context to answer questions (e.g., "What did I eat yesterday?" should reference the seed data).
7. **Demo data separation**: Log a new meal while in demo mode. Then, log in to a real account. Verify the demo meal does not appear in the real account's history.
8. **Demo reset behavior**: Trigger a demo reset (if available) or clear app data and re-enter demo mode. Verify the state returns perfectly to the initial seed baseline.
9. **Pollution prevention**: Ensure that no analytics or usage metrics from demo mode are mixed with real user analytics on the backend.
10. **First tester safety**: For initial beta testers using demo mode, ensure there are no paths that accidentally trigger real API calls intended only for authenticated users (e.g., trying to sync demo data to the cloud).

## 6. Expected pass behavior
* Demo mode provides a fully functional, read/write experience isolated entirely from real production data.
* Visual indicators constantly remind the user they are in a demo state.
* Transitioning between demo and real accounts leaves zero residual data in either direction.

## 7. Fail examples
* A workout logged in demo mode appears in a real user's account after they log in on the same device.
* Jarvis cannot answer questions about the demo seed data, stating it has no history.
* The demo dashboard is completely blank upon entry.
* Real user analytics are flooded with events generated by demo mode users.

## 8. Required notes/screenshots/logs to capture
* Capture screenshots of the demo mode indicator and seeded dashboard.
* If data pollution occurs, capture local storage dumps showing the exact keys that leaked.
* Document any console warnings related to state mismatches during mode transitions.

## 9. Blocking vs non-blocking issues
* **Blocking**: Any leakage of demo data into a real account, or real data into the demo state. Complete failure of the demo mode to load.
* **Non-blocking**: Minor visual bugs in the demo mode banner or slightly delayed rendering of demo graphs.

## 10. Future automation opportunities
* Playwright E2E tests should routinely enter demo mode, assert the presence of seed data, modify it, and then verify a real login starts fresh.
* Unit tests should strictly assert the boundary between the demo state slice and the real state slice in the store.
